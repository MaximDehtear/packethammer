Protocol overview

SimpleTCP

Line-delimited ASCII authentication protocol on TCP/2121. Clients authenticate with a username + password pair, receive a fixed session token, then issue computation commands (sum, concat, greet, multiply) — each requiring the token as a first argument.

Command set

7 commands discovered

Every command the server accepts — syntax, authentication requirement, behaviour, and any security flags identified during live instrumentation and Ghidra decompilation.

USER
Pre-auth
USER <username>

Initiates the login sequence. Username is read via std::istream::operator>> and stored in a local variable. The server always responds 331 regardless of whether the username exists — username enumeration is not possible from response codes alone.

PASS
Pre-auth
PASS <password>

Completes login. Server simultaneously checks stored username against config['login'] and supplied password against config['password'] via std::operator==. Both must match. Success: 230 TOKEN TOKEN-LOCAL-12345, session continues. Failure: 530 Authentication failed, connection closed immediately.

QUIT
Any state
QUIT

Graceful disconnect. Server responds 221 Bye and closes the TCP connection. Available both before and after authentication. Sets internal flag iVar8=1 which terminates the dispatch loop.

ST
Post-auth
ST <token> <a> <b>

Returns the integer sum of a and b as 200 ST <a+b>. Both operands parsed via std::istream::operator>> into signed int variables. Implemented at command_st (0x103fe4). No overflow concern — result formatted via std::to_string.

CT
Post-auth
CT <token> <s1> <s2>

Concatenates two strings and returns 200 CT <s1s2>. Uses std::operator+ on std::string objects — heap-allocated, no buffer overflow risk. Implemented at command_ct (0x10409a).

HI
Post-auth
HI <token> <name>

Greets with name: returns 200 HI Hello <name>. The name argument (no length limit from istream) is passed to strcpy into a 24-byte stack buffer — any name > 23 bytes overwrites the saved return address. Stack canary present but bypassable via info-leak.

⚠ Critical overflow
IV
Post-auth
IV <token> <a> <b>

Returns signed int32 product: 200 IV product=<a*b>. The product is used directly as the length argument to memset on a 64-byte scratch buffer. If a*b overflows and wraps to [0,63], memset fires with the wrapped value — unexpected but bounded. Implemented at command_iv (0x104342).

⚠ Integer overflow
Vulnerabilities found

5 security issues discovered automatically

PacketHammer's code-analyzer agent flagged these dangerous patterns by combining Ghidra static decompilation with live Frida instrumentation — every finding below was produced without a human reading a single line of assembly.

VULN-HI-BO High stack buffer overflow

Unchecked strcpy / stack buffer overflow

command_hi @ 0x1041ba

The user-supplied name argument (read via std::istream::operator>> with no length limit) is passed directly to strcpy(name_buffer, src) where name_buffer is a 24-byte stack array. Any name longer than 23 bytes overflows the buffer and overwrites the saved RBP and return address.

A stack canary is present (__stack_chk_fail @ 0x103790), providing partial mitigation — a direct overflow triggers abort. However, if an attacker can leak the canary value (e.g. through a format-string or memory read primitive), full RIP control is achievable. The overflow is reachable after successful authentication: USER admin → PASS secret → HI TOKEN-LOCAL-12345 <payload>.

Proof-of-concept trigger (crashes on stack canary check)
HI TOKEN-LOCAL-12345 AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA
VULN-RECV-DOS Medium resource exhaustion

Unbounded memory growth in recv_line

recv_line @ 0x103eff

The line-reading function at 0x103eff loops recv(fd,&ch,1,0) and appends each byte to a std::string via push_back() until '\n' is received. There is no maximum line length, no timeout, and no iteration cap.

An attacker sending a continuous stream of non-newline bytes forces the string to grow without bound. Each reallocation doubles capacity, leading to exponential memory consumption. The server process is OOM-killed by the OS. This requires no authentication and affects all concurrent sessions — a single TCP connection causes a reliable denial of service.

Proof-of-concept trigger
Connect to TCP/2121 — send continuous non-'\n' bytes until OOM kill (no auth needed)
VULN-IV-INTOVF Low integer overflow

Signed integer overflow → unexpected memset

command_iv @ 0x104342

The product __val = a * b is computed as a signed 32-bit multiplication with no overflow check. If the mathematical product overflows int32 and wraps to a value in [0, 63], the code executes memset(scratch, 0x41, (long)__val) on a 64-byte scratch buffer — writing exactly __val bytes of 'A'.

No out-of-bounds write occurs (scratch is 64 bytes, __val < 64), but the unexpected control flow may interact with adjacent memory depending on stack layout. Exploiting this requires engineering a*b mod 2³² ∈ [0,63] — mathematically constrained but achievable with chosen large operands.

Proof-of-concept trigger
IV TOKEN-LOCAL-12345 65536 65536
VULN-UAF Info use after free

Use-after-free (suspected — not confirmed)

handle_client @ 0x1043aa

Ghidra's decompilation of handle_client (0x1043aa, 2637 bytes) shows manual destructor call sequences — basic_string() constructor, send_line(), ~string(), ~__new_allocator() — interleaved with complex branch chains. If send_line() throws or control flow is interrupted mid-sequence, a string object could be left in an inconsistent state.

Most likely this is a Ghidra decompilation artefact from inlined std::string code — actual RAII semantics in the binary are likely intact. Not confirmed exploitable. Recorded because the pattern warrants fuzzing attention, particularly under exception-triggering conditions.
